# Repo Intake Note

このスキルは、ローカルリポジトリを既定で読み取り、必要時のみユーザー指定の Git リポジトリも対象にして、内容を把握しやすい Markdown に整理する時に使う。

## Use When

- リポジトリの全体像を短時間で把握したい
- README、docs、設定、コード、テストから内容を棚卸ししたい
- 要件、仕様、設計、実装、挙動、知見を分けて整理したい
- 他案件にも転用できる知見を抽出したい

## Input Handling

- ユーザーがローカルパスを指定した場合は、それを最優先で対象とする。
- ユーザーが対象を指定していない場合は、現在の作業ディレクトリまたは明示的に参照可能なローカルリポジトリを対象候補とする。
- ユーザーがリポジトリ URL、`owner/repo`、または「Git リポジトリを読んで」と明示した場合のみ、Git リポジトリを対象とする。
- Git リポジトリの明示指定がない限り、`git clone` やリモート参照を前提にしない。
- ユーザーが注目テーマを指定した場合は、その観点を優先して読む。
- 対象が大規模な場合でも、最初に README、docs、設定ファイル、ディレクトリ構成を確認する。
- 明示されていない広範囲な別ディレクトリは自動で読みに行かない。

## Definitions

- 要件: 何を実現する必要があるか。対象ユーザー、目的、課題、満たすべき条件
- 仕様・制約: システムがどう振る舞うか。画面、API、入出力、制約、業務ルール
- 設計・構成: どういう構造で実現しているか。アーキテクチャ、責務分担、主要モジュール、データ構造
- 実装の要点: 実際にどう作られているか。主要クラス、関数、依存関係、設定、実装パターン
- 確認できる挙動: 実行時やテストから確認できる動作。処理フロー、状態遷移、エラー時の動き

## Workflow

1. まず対象がローカルリポジトリか、ユーザー明示の Git リポジトリかを判定する。
2. ローカル対象ならそのまま読み始め、Git 対象なら必要な取得方法を確認したうえで読む。
3. README、docs、設定ファイル、ディレクトリ構成を確認する。
4. 主要なエントリポイント、主要モジュール、テストを確認する。
5. 要件、仕様・制約、設計・構成、実装の要点、確認できる挙動を分けて整理する。
6. リポジトリ固有の情報と、横断的に再利用できる知見を分ける。
7. 事実には根拠ファイルを付け、推測は `要確認` と明記する。
8. まずは 1 ファイルにまとめる。
9. 必要な場合のみ、親ファイルを残したまま詳細ファイルへ分割する。

## Output Format

以下の見出し順を基本とする。

- タイトル
- 概要
- 読み取った範囲
- 要件
- 仕様・制約
- 設計・構成
- 実装の要点
- 確認できる挙動
- ディレクトリ / モジュール構成
- 知見
- 要確認事項
- 参照ファイル

## Writing Rules

- 既存文書の丸写しは避け、要点を再構成する。
- できるだけ簡潔に書く。
- 重要な内容から先に書く。
- 事実には根拠ファイルを添える。
- 推測を含む内容は必ず `要確認` と明記する。
- デフォルトでは 1 ファイルにまとめる。
- 分割する場合は、要約用の親ファイルを 1 つ作り、詳細ファイルへの導線を付ける。
- 知見では、リポジトリ固有の話と横断知見を混在させない。

## Output Location

保存先が求められた場合の第一候補は以下とする。

- `docs/99_tmp/YYYYMMDD_<repo>_intake.md`

必要に応じて以下へ分割してよい。

- `docs/99_tmp/YYYYMMDD_<repo>_requirements.md`
- `docs/99_tmp/YYYYMMDD_<repo>_architecture.md`
- `docs/99_tmp/YYYYMMDD_<repo>_knowledge.md`
