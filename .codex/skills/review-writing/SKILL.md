# Review Writing

このスキルは、Markdown 文書の文体や書き方をこのリポジトリの運用方針に合わせて整える時に使う。

## Use When

- ユーザーが文体レビュー、文章の簡潔化、Markdown の表現改善を求めている
- 誇張表現や AI っぽい言い回しを減らしたい
- 指摘と合わせて短い改善案を出したい

## Input Handling

- ユーザーが対象ファイルを明示した場合は、そのファイルを元情報として最優先で読む。
- 想定対象は `docs/**/*.md` `README.md` `.claude/**/*.md` とする。
- ファイル指定がない場合のみ、会話内の文章をレビュー対象として扱う。
- 複数ファイルがある場合は、表現ルールの揺れや粒度の差も確認する。
- 明示のない `docs/99_tmp/` 全体を自動走査せず、指定されたファイルか会話内の入力だけを使う。

## Workflow

1. 過剰な強調、誇張表現、AI っぽい定型句がないか確認する。
2. 自明な説明、重複説明、冗長な言い回しを抽出する。
3. 推測や未確認事項に `要確認` が付いているか確認する。
4. 見出し、箇条書き、表現の粒度がそろっているか確認する。
5. 問題箇所ごとに、短く自然な修正文を示す。

## Output Format

以下の項目順を基本とする。

- 対象
- 指摘事項
- 改善案
- 要確認事項

## Writing Rules

- 指摘は文体と表現に絞り、事実確認とは分ける。
- 改善案は短く自然な日本語を優先する。
- リポジトリ方針に反する場合は、その観点を明記する。
- 対象ファイルがある場合は、その内容を正本の入力として優先する。
